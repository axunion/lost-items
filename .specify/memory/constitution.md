# lost-items Constitution

## Core Principles

### I. 厳格な品質と一貫性 (Strict Quality and Consistency)
コードの品質と一貫性を厳格に保ちます。プロジェクト内のディレクトリ名、ファイル名、およびコード内の命名規則は、定められた規約に従い、適切に統一されなければなりません。
(Maintain strict code quality and consistency. Directory names, file names, and in-code naming conventions within the project must be appropriately unified according to established conventions.)

### II. 言語方針 (Language Policy)
開発プロセス中に作成される内部向けドキュメント（設計書、議事録など）は日本語で記述します。しかし、`README.md`、コード内のコメント、エラーメッセージ、APIレスポンスなど、プロダクトの成果物として外部に公開される、あるいは開発者間で共有されるテキストは英語で統一します。
(Internal documents created during the development process (designs, minutes, etc.) will be written in Japanese. However, text that is published externally as a product artifact or shared among developers, such as `README.md`, comments in code, error messages, and API responses, will be standardized in English.)

### III. 効果的なコメント (Effective Commenting)
自明なコードに対するコメントは避け、コードの意図や複雑なロジックの「なぜ」を説明するためにのみコメントを使用します。コメントはシンプルかつ適切に、必要な場合にのみ追加してください。
(Avoid comments for self-evident code. Use comments only to explain the "why" behind code intent or complex logic. Comments should be simple, appropriate, and added only when necessary.)

### IV. コミットメッセージの規約 (Commit Message Style)
コミットメッセージは、変更内容が明確にわかるように、規約に定められた適切なprefix（例: `feat:`, `fix:`, `docs:`）を付与し、シンプルな英語で記述します。
(Commit messages must be written in simple English with an appropriate prefix as defined in the conventions (e.g., `feat:`, `fix:`, `docs:`) to clearly convey the nature of the change.)

## Governance
This constitution supersedes all other practices. Amendments require documentation and approval. All contributions and reviews must verify compliance with the constitution.
